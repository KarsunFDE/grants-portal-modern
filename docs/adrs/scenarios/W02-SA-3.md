# ADR W02-SA-3: Embedding Model Selection for RAG Layer

**Status:** Accepted  
**Date:** 2026-06-04  
**Deciders:** Ronald Forte (pair)  
**Scenario:** W02-SA-3 — Embedding model selection: Titan Embeddings v2 vs domain-tuned vs embedding-routing

---

## Decision

**AWS Bedrock Titan Embeddings v2 (`amazon.titan-embed-text-v2:0`) as the single embedding model for all content types** — FAR/DFARS regulatory clause text and vendor CPAR narratives alike.

Embedding-routing (split by content type) is deferred, not dismissed. If the Friday W2 eval harness shows a >10% NDCG@10 gap in CPAR retrieval favoring Cohere Embed v3, routing adoption is pre-approved with the swap cost accepted (see Rollback Story).

---

## Context

The RAG layer must index two structurally different content types:

- **FAR/DFARS clause text** — formal regulatory English, consistent structure, dense cross-references, bodies routinely exceeding 1,000–3,000 words (e.g., FAR 52.215-1).
- **CPAR narratives** — free-form English, written by Program Managers, variable vocabulary, domain jargon mixed with informal prose.

Production load estimate: ~70% clause text, ~30% CPAR narratives. Corpus: ~25,000 chunks.

Three alternatives were evaluated. No public FAR/DFARS-specific embedding benchmark exists at time of decision. The Friday W2 eval harness will produce domain-specific retrieval metrics for the first time.

Constraint: All model invocations must remain within the FedRAMP boundary. Both Titan v2 and Cohere Embed v3 are available on AWS Bedrock and are in-boundary.

---

## Alternatives Considered

### Option A — Titan Embeddings v2 Only (Selected)

AWS Bedrock model `amazon.titan-embed-text-v2:0`. General-purpose English embedding model, second-generation. Unit-normalized output (cosine similarity equals dot product).

| Dimension | Score (1–5) | Notes |
|---|---|---|
| Capability fit | 3 | Strong English; 8,192-token input handles full clause bodies; no `input_type` param — symmetric embeddings only |
| Cost shape | 5 | $0.02/1M input tokens flat across all content types |
| Operational burden | 5 | Single Bedrock endpoint, single vector index, single deprecation surface, zero router code |
| Quality (pre-harness) | 3 | Optimized for English; no FAR/DFARS-specific public eval; same "general model" tier as Cohere v3 |
| FedRAMP boundary | 5 | Fully in-boundary on Bedrock |
| W5 AIOps dial | 3 | Single cost dimension; simpler but less granular than per-content-type split |
| **Total** | **24/30** | |

**Why selected:** Highest total score. The decisive factor is what Option B and C cannot offer — Titan v2's 8,192-token input limit is a direct architectural fit for long-form clause text. Operational simplicity at W2 avoids premature complexity when no quality evidence yet justifies routing overhead.

---

### Option B — Cohere Embed v3 Only (Rejected)

AWS Bedrock model `cohere.embed-english-v3`. Provides `input_type` parameter (`search_document` / `search_query`) enabling asymmetric search — query and document embeddings are optimized differently, which is theoretically beneficial for free-form narrative retrieval.

| Dimension | Score (1–5) | Notes |
|---|---|---|
| Capability fit | 3 | `input_type` asymmetric search is a genuine feature; 512-token hard limit per text is a critical problem for FAR clause bodies |
| Cost shape | 1 | $0.10/1M tokens — 5× more expensive than Titan v2 with no proven quality lift for regulatory text |
| Operational burden | 4 | Single model, but 512-token limit forces more aggressive chunking and chunk-overlap strategy for clause text |
| Quality (pre-harness) | 3 | MTEB English retrieval: 55.0% (general benchmark only); domain-tuned legal models (voyage-law-2) outperformed Cohere v3 by 6%+ on 8 legal datasets — Cohere v3 is not a domain-specialized model |
| FedRAMP boundary | 5 | Fully in-boundary on Bedrock |
| W5 AIOps dial | 3 | Single cost dimension, same as Titan v2 |
| **Total** | **19/30** | |

**Why rejected:** Worst cost score (1/5) with no evidence of quality advantage over Titan v2 on regulatory English. The 512-token per-text limit is a hard architectural constraint against FAR clause text. Paying 5× for an unproven benefit is indefensible at W2.

---

### Option C — Embedding-Routing: Titan v2 for Clauses, Cohere v3 for CPARs (Deferred)

Application-side routing using `content_type` metadata. Clauses are embedded with Titan v2 (`amazon.titan-embed-text-v2:0`); CPAR narratives are embedded with Cohere v3 (`cohere.embed-english-v3`). Routing occurs at ingestion and at query time. Because models produce incompatible vector spaces, two separate vector indexes must be maintained.

Weighted cost: `($0.02 × 0.70) + ($0.10 × 0.30) = $0.044/1M tokens` — 2.2× Titan v2 flat cost.

| Dimension | Score (1–5) | Notes |
|---|---|---|
| Capability fit | 4 | Best model per content type in theory; Cohere `input_type` helps CPAR queries; Titan v2's 8,192-token limit covers clause bodies |
| Cost shape | 3 | $0.044/1M weighted average — 2.2× overhead vs Titan flat; justified only if quality delta confirmed |
| Operational burden | 2 | Router function (must be tested, must fail loudly on new content types), dual vector indexes, dual Bedrock model endpoints, dual deprecation surfaces |
| Quality (pre-harness) | 3 | Routing is a hypothesis — no evidence Cohere v3 outperforms Titan v2 on CPAR narratives; voyage-law-2 data shows both are general-purpose tiers; routing quality lift unproven |
| FedRAMP boundary | 5 | Both model legs in-boundary on Bedrock |
| W5 AIOps dial | 4 | Per-content-type cost dimensions; more granular and actionable for W5 governance |
| **Total** | **21/30** | |

**Why deferred (not rejected):** The asymmetric search argument for CPARs is architecturally sound. The operational complexity and cost premium are justifiable *if* Fri eval confirms >10% retrieval quality lift. Deferred to post-Fri-eval re-evaluation with a named adoption threshold.

## Consequences

### What we gain (Titan v2 only)
- Lowest cost ($0.02/1M flat) — cleanest W5 AIOps cost line
- Single vector index — unified retrieval across clause and CPAR corpus in one similarity search
- 8,192-token input — handles full clause bodies without aggressive chunking
- Zero router code to test, maintain, or monitor for silent failures
- Single deprecation surface — one model version migration if Bedrock deprecates

### What we give up (Titan v2 only)
- Asymmetric search for CPAR narratives (`input_type` query/document distinction)
- Per-content-type cost observability at W5 AIOps (only recoverable by adding metadata tagging to Titan v2 invocations at ingestion — possible without model change)
- Any quality lift routing might provide if Fri eval confirms Cohere v3 wins on CPARs

### Risks
- If Fri eval shows significant quality regression on CPAR retrieval vs Cohere v3, adoption of routing requires re-embedding 30% of corpus (~7,500 CPAR chunks) plus building the router and dual-index infrastructure — multi-hour batch, not a hotfix.
- Titan v2 has no public MTEB scores on the general English leaderboard (as of 2026-06-03). If MTEB benchmarking is required as evidence, Cohere v3's published score (55.0% English retrieval) gives it a documentation advantage — not a quality advantage.

---

## Rollback Story

**Current pick regresses on Fri eval (NDCG@10 gap >10% in CPAR retrieval favoring Cohere v3):**

1. Switch CPAR ingestion pipeline to use `cohere.embed-english-v3` with `input_type=search_document`
2. Re-embed ~7,500 CPAR chunks (30% of ~25K corpus) via batch job — estimated 2–4 hours depending on throughput quota
3. Create second Pinecone/Atlas namespace for Cohere v3 CPAR vectors
4. Write and test router function (~1 file, content_type metadata → model + namespace selector)
5. Update query path to route CPAR queries to Cohere v3 namespace with `input_type=search_query`

**Estimated swap cost:** 1 engineer-day of implementation + 2–4 hours batch re-embedding. Tractable. Not a blocker.

**Current pick regresses on Fri eval (Titan v2 underperforms on clause retrieval vs any alternative):**
All 25K chunks must be re-embedded. Multi-hour batch. Plan for off-hours execution.
